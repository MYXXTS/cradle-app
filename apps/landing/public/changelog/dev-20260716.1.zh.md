> 让想法开始生长，也让工作真正抵达。

Chat 适合探索，Work 用来完成。

一段对话可以帮助你理解问题、尝试方向或者修改代码，但真正的工作往往更长：它有一个明确目标，需要独立的环境，会经历执行、等待、审查和返工，最后还要形成 Commit 或 Pull Request。

这次更新带来了 **Cradle Work**——一个承载这段过程的新产品概念，以及围绕它重新构建的工作台、上下文、资源管理和交付体验。

## ✨ Cradle Work

Cradle Work 不等于一段聊天，也不依附于 Issue 或 Kanban。你可以随时创建一个 Work，写下想完成的事情，然后选择合适的 Agent、模型和执行环境。Issue 和 Kanban 是可选的上下文入口，而不是 Work 存在的前提。

一个 Work 会记住目标、工作环境、相关 Session、当前进度与交付状态。对话可以暂停、继续或交接，但工作本身不会因此消失。

`目标 → 隔离执行 → 检查改动 → 本地 Commit → 用户确认 → Pull Request`

Agent 会先在隔离环境中工作。完成后，Cradle 将改动带回可审查的界面，让你决定是否继续修改、创建本地 Commit，或进入交付流程。Push 和创建 Pull Request 等远程操作会在得到明确确认后才发生。

Work 也开始真正出现在 Cradle 的日常界面中：它与相关 Session 一起呈现在 Workspace 侧边栏，并统一显示运行、等待、阻塞、Draft PR、Ready PR 和已合并状态。

这不是一种新的聊天模式，而是一种与 Agent 一起完成工作的方式。

## 📥 把过去的工作带回来

新的 **Import Center** 可以发现本机或远程服务器上的 Claude Code 与 Codex 会话。

Cradle 会还原它们原本所属的项目，展示可恢复的 Workspace，并让你选择需要带回来的对话。导入完成后，这些记录不再只是散落在不同 Provider 目录中的 JSONL 文件，而会成为可以搜索、浏览、继续和归档的 Cradle Session。

已经导入的会话还可以继续同步。Cradle 会识别新增消息，也会在源历史被改写时停止猜测，避免用不可靠的方式合并两段已经分叉的记录。Claude 与 Codex 的 Subagent 历史也会作为主会话的一部分保留下来。

导入过程始终保持外部数据只读。Cradle 会先创建经过大小与 SHA-256 校验的自有 Bundle，再从这份副本生成聊天记录；Claude 和 Codex 原本的数据目录不会被修改。

## 🧭 Workspace 变成了一张真正的工作台

Workspace 不再只是 Session 的分组方式，而开始承载实际工作的各个表面。

你可以在右侧打开浏览器查阅文档、查看部署或操作 Web 工具，也可以从底部打开 Terminal 和 TUI。面板会跟随 Workspace 生命周期，在关闭后自动释放相关资源。

远程 Workspace 也可以被浏览和导入。远程 Session 会投影到本地界面，并在 Host 离线时明确阻止新的操作，而不是让消息静默失败。

如果项目目录被移动、磁盘暂时离线，Cradle 会将 Workspace 标记为缺失，并允许重新关联到新的位置。

这次更新还允许迁移 Cradle 自己的数据目录。数据库、日志、插件与 Runtime 文件可以移动到其他磁盘。迁移会经过临时复制、逐文件校验和新 Server 健康检查；只有新位置确认可用后才会切换，旧目录则作为备份保留。

## 📦 让工具和环境也成为产品的一部分

新的 **Resources** 页面用于管理 Cradle 运行所依赖的模型资源和 Runtime。

OpenCode 可以作为托管 Runtime 安装、升级和切换版本；应用更新、模型、插件与 Runtime 下载则统一进入 **Download Center**。下载任务拥有一致的进度、取消、重试与完整性校验，并能在页面关闭或应用重启后保留状态。

Desktop 现在会附带独立的 Codex App Server，减少对外部 Codex 安装环境的依赖。需要与原生 CLI 保持一致时，还可以选择使用 Codex CLI-compatible Identity。

Plugin Center 获得 Marketplace、已安装插件和本地导入页面。插件开发者还可以通过 CLI 启动 Development Session，让 Desktop 自动同步并热更新正在开发的插件。

## 💬 从对话走向连续的上下文

Composer 现在更适合处理真实工作中的复杂输入。

长段粘贴文本会变成独立卡片，可以预览、展开、恢复到编辑器或移除。你也可以添加上下文附件，引用文件和具体代码行，而不必把所有信息塞进一条纯文本消息。

Agent 运行期间可以继续追加 Follow-up，并编辑或调整等待中的指令。上一条消息也可以被重新编辑和执行，同时保留 Workspace 中已经产生的文件改动。

当模型无法直接读取图片时，本地 Light OCR 可以先提取文字。Mermaid 和受控 HTML 也能直接在 Chat 中渲染，让架构图、流程图和技术说明不再只是代码块。

Environment Handoff、Turn Checkpoint、Session Group 和 ZIP Export 则让长时间工作可以暂停、恢复、整理和交接。

## 📊 更准确地理解 Agent 的消耗

Usage 页面现在会同时呈现两种视角：一边是 Claude 与 Codex 本地 Archive 中记录的设备级权威总量，另一边是 Cradle 可以直接归因到具体运行和 Session 的活动。

主 Session 与 Subagent 的消耗会聚合为完整的 Session Tree。之前遗漏的 Codex 模型调用也会自动回填，并避免重复计算已有数据。

你可以按时间、模型和 Session 查看 Prompt、Completion、Cache、Reasoning Token 与估算成本，理解一次完整工作真正消耗了什么。

## ⚡ 不那么显眼，但很重要的变化

这次更新重新设计了 Chat 的长期存储方式。

消息正文不再被重复保存到每一条事件中，而是作为独立 Payload 存储和按需读取。已完成运行产生的大型诊断快照会在后台自动压缩，过期快照也会按照保留策略清理——这些维护只针对运行诊断数据，不会删除正常聊天记录。

Streaming 使用单调 Cursor 进行增量同步，重连时更不容易重复或遗漏输出。Work Context 也从频繁变化的 System Prompt 中分离，使 Agent 在长任务中更稳定，同时提高 Prompt Cache 的复用率。

Diff Review Guide、Commit Plan 和其他耗时流程开始由持久化 Background Jobs 承载。即使应用重启，Cradle 仍然知道任务执行到了哪里，以及结果应该回到哪个产品界面。

OpenCode Runtime 现在按 Workspace 复用，减少重复进程和启动成本；Renderer 也会等待本地 Server 就绪后再加载主要模块，降低启动竞态和无效请求。

## 💎 还有这些改进

- 新增 Pull Request 仪表盘、详情页、Review 状态与大规模游标分页
- Session 可以创建 Pull Request、标记 Ready，并等待 CI 或 Review 结果
- 每个 Provider 和 Model 可以独立记忆 Thinking Effort
- 新增 `none`、`minimal`、`max` 与 `ultra` 推理级别
- Provider 可以展示自己的权限、交互模式与 Plan Mode 设置
- 新增主题定制、终端字体、首选编辑器、快捷键和更新偏好
- Provider 配置导入支持手动解码 Base64 API Key，并可以随时恢复原值
- Runtime 的可重试错误会显示在时间线中，而不会直接中断回复
- Claude Agent 的工具活动、告警、等待状态与 Usage 会显示为结构化界面
- Session 支持快速预览、一键全部标为已读和自定义分组
- Windows、Linux 与 macOS 获得统一的自定义标题栏

## 🔐 安全与隐私

- Import Center 只读取 Claude 与 Codex 的数据目录，不会修改外部 Provider 的文件
- 下载任务会验证文件大小与 checksum，降低损坏或来源不可信的资源被安装的风险
- 不再安装缺少可信 checksum 的 ACP 二进制包，目前仅提供 `npx` 与 `uvx` 分发方式
- 敏感凭证改由运行时 Keyring 统一管理
- 收紧本地、远程 Runtime 和 WebSocket 的认证与生命周期边界
- 新增匿名功能使用分析与 AI Observability；失败分类不会发送原始错误文本

- 修复 Browser Panel 的资源泄漏、初始化竞态、快捷键冲突和 Preload URL

- 修复多轮 Rollback、队列与 Synthetic Turn 的状态冲突
- 修复 OpenCode 工具结果和 Todo 项的结构化显示
- 修复 Git 提交图显示内部 `refs/cradle/*` 引用
