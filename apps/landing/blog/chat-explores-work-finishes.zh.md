> Chat is for exploration. Work is for finishing.

一段对话可以帮你理解一个问题、探索一个方向、改动一些代码。但真正的工作通常活得更久：它有明确的目标，需要自己的环境，会经历执行、等待、评审、修改，最终要变成一个 commit 或一个 pull request。

对话的形态承载不了这个过程。对话结束了，工作还在；窗口关掉了，事情没完。这就是我们引入 **Cradle Work** 的原因。

## Work 不是另一种聊天

Cradle Work 不依赖 issue，也不依赖看板。你随时可以创建一个 Work，描述你想完成什么，选择合适的 agent、模型和执行环境。Issue 和看板是可选的上下文来源，而不是前置条件。

一个 Work 会记住自己的目标、工作环境、关联的会话、当前的进度和交付状态。对话可以暂停、继续、被交接，而工作本身不会因此消失。

## 一条清晰的交付路径

`目标 → 隔离执行 → 评审变更 → 本地提交 → 用户确认 → Pull Request`

Agent 先在隔离的环境里工作。完成之后，Cradle 会把变更带回一个可评审的界面，由你决定是继续修改、创建本地 commit，还是进入交付流程。推送、开 PR 这类远端动作，只会在你明确确认之后发生。

这条路径里最关键的设计，是**每一步的边界都属于你**。Agent 负责执行，你负责决定什么算完成。

## 为什么这很重要

AI 编程工具大多把力气花在"开始"上——怎么更快地生成代码。但软件工作的瓶颈从来不在开始，而在完成：评审、返工、等待、交付。

Work 把这一整段过程变成一个一等公民。它有状态（running、waiting、blocked、Draft PR、Ready PR、merged），有记忆，有边界。你不再是盯着聊天框等结果的人，而是站在交付路径终点做决策的人。

这不是一个新的聊天模式。这是一种和 agent 一起把事情做完的方式。
